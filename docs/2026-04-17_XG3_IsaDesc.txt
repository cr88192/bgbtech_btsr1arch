Behavioral Spec for XG3.
* Will focus on the Userland ISA.
* XG3 May co-exist with RV64G in the same encoding space.

Overview:
* XG3 is an ISA derived from the BJX2 ISA.
* It may be used alongside the RISC-V ISA within a shared operating mode.
* Encodings are variable length, with 32, 64, or 96 bit instruction encodings.

Premise:
* XG3 is not intended as a full replacement for RV64G or RV64GC.
* Rather as a Special Purpose ISA intended for performance oriented use cases.
* Typical interaction between ISA modes will be at DLL or SO boundaries.


== General ==

XG3 is incompatible with RV-C in that both may not be used at the same time, however a processor may support both and it is possible in theory to use both within a single binary.

Note that XG3 will be described in terms of an AT&T style ASM notation, with the destination on the right hand side. The choice between AT&T and RISC-V style notation and nmemonics will be up to the compiler and assembler. This ISA may alternatively be expresssed using RISC-V inspired conventions.


=== Encodings ===

General Encoding Patterns:
* XXXX-oooooo-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (3R)
* jiii-iiiiii-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (3RI Imm10/Disp10)
* iiii-iiiiii-iiiiii-aZZZ-nnnnnn-bY-YYPw (2RI Imm16)
* iiii-iiiiii-iiiiii-aZZZ-jjjjjj-bY-YYPw (BRA/BSR)
* iiii-iiiiii-iiiiii-jjjj-jjjjaa-a1-1Y10 (Jumbo Prefix)


Pw:
* 00: OP?T (Conditional True, Optional)
* 01: OP?F (Conditional False, Optional)
* 10: OP   (Unconditional)
* 11: RV64G Space

Where, YYY:
* 000: F0 (3R ops)
* 001: F1 (LD/ST Disp10)
* 010: F2 (3RI Imm10 Ops)
* 011: F3 (Reserved / User, 3R Ops)
* 100: F8 (Imm16 ops)
* 101: F9 (Reserved, 3R Ops)
* 110: FE (Jumbo Prefix, J_IMM)
* 111: FF (Jumbo Prefix, J_OP )

Exact repacking rules will depend on the encoding block.
* F1/F2/F8: Will modify repacking such that immed bits are contiguous.
* F3 and F9 will use same repacking as F0.

Where:
* nnnnnn: Destination Register
* mmmmmm: First Source Register
* oooooo: Second Source Register
* iiiiii: Immediate / Displacement

For typical Imm10/Disp10 encodings, the Imm/Disp field will be a contiguous value, with variation for sign extension:
* Imm10u: Value is Zero Extended  (0     .. 1023)
* Imm10n: Value is One Extended   (-1024 ..   -1)
* Imm10s: Value is Sign Extended  (-512  ..  511)

Typically, the Disp10 fields will be scaled based on the instruction.
* Load/Store by Element Type
* Branches by 4.


Note that instruction words with the low two bits set will map to the RISC-V encoding space. XG3 and RV64G instructions may be freely mixed, except that in XG3 Mode, the RISC-V instructions will be understood in a way consistent with XG3 semantics rules. This may not necessarily be true in RV64GC Mode, with the partial exception that branching to an address with the LSB set may be used to transfer control between modes.


=== Registers ===

User Visible Registers:
* R0 / ZR / ZERO          : Zero Register
* R1 / LR / RA            : Link Register
* R2 / SP                 : Stack Pointer
* R3 / GP                 : Global Pointer
* R4 / TP                 : Task Pointer
* R5  .. R7  / X5 .. X7   : Scratch / Stomp
* R8  .. R9  / X8 .. X9   : Callee Save
* R10 .. R17 / X10.. X17  : Scratch / Arg1..Arg8
* R18 .. R27 / X18.. X27  : Callee Save
* R28 .. R31 / X28.. X31  : Scratch
* R32 .. R35 / F0 .. F3   : Scratch
* R36 .. R41 / F4 .. F9   : Callee Save
* R42 .. R49 / F10.. F17  : Scratch / Arg9..Arg16 (Optional)
* R50 .. R59 / F18.. F27  : Callee Save
* R60 .. R63 / F28.. F31  : Scratch


Architectural Control Registers:
* PC: Program Counter
* SR: Status Register
* GBH: Data Quadrant
* PCH: Program Quadrant

* TBR: Task Base Register
* FPSCR: FPU Status/Control

These architectural registers may have an indirect effect on the behavior of the program, but may not be directly visible nor addressible within the program itself.


Note that for 128-bit ALU operations, X0 may be special:
* X0 will be treated as a 128-bit Zero Register.
* Note that X0/X1 and X2/X3 will not pair.
** For ALU or SIMD: X1..X3 are undefined.
** For XMOV: They may imply expanded registers.
*** R2 = GBH:SP
*** R3 = GBH:GP
* For 128-bit operations, only even-numbered register pairs are allowed.


=== Address Space ===

Nominally, XG3 will use an address space with a 48-bit address and 16 tag bits. For Load/Store operations, these tag bits are to be ignored.

For instructions using a 96-bit address, the low 48 bits of each register in the pair will be understood as part of the address, with the high 16 bits of each register as tag bits. As before, in typical cases, the tag bits are ignored.

There may be optional support for hardware bounds checking, but this will be handled by explicit instructions rather than built into the Load/Store instructions.

Note that in this case, the current 48-bit space will exist "somewhere" within the 96 bit space, with the architectural GBH and PCH registers defining the location within the wider address space. For sake of naming, each 48 bit space (roughly 1/4 of a petabyte) will be referred to as a "Quadrant" or "Quad".

Typically, Code and Data will exist within the same Quadrant, but these may differ, in which case the processor may appear like a Harvard architecture (rather than taking on the more traditional Von Neumann appearance when equivalent).

The extent to which 96-bit addressing is supported natively will be implementation dependent.

Note that access within the Local 48-bit quadrant may be considered local and may allows for (relatively) unrestricted address, but the larger 96-bit space may operate within a Capability Addressing scheme.


Within the Capability Scheme, the remaining 32 tag bits will be used to encode the relative address bounds (for the lower 48-bit address). If an access is out of range for a capability (or to an address that is not a valid capability, in BCE Mode), the access will fault. Note that if BCE is not supported or not enabled, then the 96 bit address will be accessed directly if Addr96 is supported by the target, otherwise a fault will be raised (for possible emulation in software).

The specifics of the capability addressing scheme and MMU will be considered outside the scope of this spec.


When used as a Base Register, R0 and R1 are special:
* R0: PC for Disp10s / Disp33; Implicit scale of 1 or 4.
** R0 is reserved for Abs64.
* R1: GP (Unscaled) for Disp6s / Disp10s, ZR for Abs33 / Abs64
** For Disp33 or scaled displacement, R3 is to be used instead.

As an Index Register:
* R0..R3 are Reserved.


Note that the Disp10s field for Load/Store will normally be scaled by the element size.

For a Disp33s encoding, the MSB of the Load/Store instruction will instead effect scale:
* 0: Disp33 is scaled by element size.
* 1: Disp33 is unscaled.


=== Link Register ===

Implicitly, PC and LR will have captured Status/Control bits in the high order bits (63:48):
* (63:56): U-Bits (Contextual Tag bits, or Fixed PC Tag)
* (55:50): Processor ISA Mode
* (49:48): S and T bits.

The T (SR.T) bit will be used primarily to control predication.
* If an instruction is Predicated True, it only runs if T is Set.
* If an instruction is Predicated False, it only runs if T is Clear.

The S (SR.S) bit will be more niche, but may be used for certain instructions.


For captured Link Register values, the LSB will always be Set.
* If branching to a register, if the LSB is Set, the high bits are valid.
* If branching to a register, if the LSB is Clear, the high bits are ignored.

If branching to LR (via JALR/etc), some additional rules may apply:
* The LSB must always be set;
* If used as a Fixed Tag, the U bits in PC and LR must match.
** A mismatch in the U bits may be treated as an architectural fault.
* The Mode will be copied from LR to PC, changing ISA Mode as needed.

For branching to other registers the rules may be relaxed:
* If LSB is Clear, the Jump is within the current Mode.
* If LSB is Set, the Jump may change modes.
** If used as a Fixed Tag, U bits are ignored if 0.
** If in Fixed Tag mode with non-zero and non-matching U-Bits, Fault.


The assumption for Fixed Tag U-Bits is that they will be assigned a random value at some point (such as on boot or process creation), and will be used to add basic protection against branching to link register values from outside of the current process.

Currently, Modes will be defined:
* 000000: Implementation Dependent / Disallowed
* 000001: Implementation Dependent / Disallowed
* 000010: RV64GC Mode
* 000011: XG3 Mode
* Others: Reserved

An attempt to branch into one of the disallowed modes from within XG3 Usermode should be regarded as an architectural fault.


Note that within XG3 Mode, some additional rules will apply to the RISC-V JAL and JALR encodings:
* Only X0 and X1 are natively allowed for the Destination Registers.
* JALR to X1 will be special, as above
* JALR to X0, or X2..X3, may be regarded as a Fault.
* JALR to X4..X31 will use the Relaxed Modes.

JAL and JALR with other values for Xd may be allowed for RV64GC Mode, but should ideally produce untagged values. The result of using these encodings in XG3 Mode will be implementation defined.

Note that RV64GC mode will be allowed to have a 16-bit instruction alignment, but within XG3 Mode, only 32-bit alignment is allowed. Attempting to branch to a location which does not have 32-bit alignment is to be regarded as a fault.



=== Jumbo Prefixes ===


Prefixes
* iiii-iiiiii-iiiiii-jjjj-jjjjaa-a1-1010 (Jumbo_FE / J_IMM)
* ZZZZ-jjjjii-iiiiii-ZZZZ-ZZZZaa-a1-1110 (Jumbo_FF / J_OP)

The J_IMM prefix will serve the primary role of expanding the immediate field.

The J_OP prefix will have a lesser role in immediate, but is more intended to expand the instruction encoding space.

Note that for sake of decode and execute, the prefix and the suffix instruction will be treated as a single instruction. It will not be possible to execute a prefix independent of a suffix nor will it be possible for an exception to occur between a prefix and suffix. Fetch/Execute will also operate on the prefixes and suffix as a singular unit.

In the Listings, prefixes may be given before the encoding:
* J_IMM: Single J_IMM prefix
* J_OP: Basic J_OP prefix
* J2_IMM: Pair of J_IMM prefixes (for 96-bit encodings).


==== Jumbo, F0 Block ====

F0/F3/F9 (J_IMM)
* iiii-iiiiii-iiiiii-jjjj-jjjjbb-b1-1010 (FE)
* XXXX-Sakkkk-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F0)

For the F0/F3/F9 block, a 33-bit immediate will be formed:
* S-bbba-jjjj-jjjj-iiii-iiii-iiii-iiii-kkkk

Where:
* Imm6u/Disp6s will be extended to Imm33s/Disp33s.
* Some 3R instructions may synthesize an immediate field.


The J_IMM prefix will convert normal 3R ops into 3RI (Imm33s).
* It will also convert 2R ops to 2RI (Imm33s, Rn).
* Behavior will typically be the same as the basic instruction.

Exceptions to J_IMM synthesizing an Imm33s may exist where the base instruction would not make sense as an Imm33s form.


The behavior of an FF prefix with an F0 instruction will depend on the F0 instruction.


F0, 4R (J_OP_4R / J_OP_R):
* UUUU-1ttttt-ssssss-0000-WWWWWW-R1-1110 (FF)
* XXXX-oooooo-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F0)

F0, 3RI Imm17s (J_OP + Imm6):
* UUUU-0jjjjj-jjjjjj-0000-WWWWWW-R1-1110 (FF)
* XXXX-Skiiii-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F0)

F0, 3R to Imm17s Synthesis (J_OP_IS / J_OP_I):
* UUUU-0jjjjj-jjjjjj-1000-WWWWWW-R1-1110 (FF)
* XXXX-Skiiii-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F0)


For the F0 block, a Imm17s may be formed if used with an Imm6 or Disp6 op:
* S-kjjj-jjjj-jjjj-iiii

With the R/U/V/W bits as additional opcode or context dependent fields.

Note tat with a J_OP prefix, any unused bits are to be treated as MBZ.


The J_OP_4R prefix will expand certain 3R ops to 4R.
* Rm, Ro, Rn => Rm, Ro, Rp, Rn
The J_OP_4R prefix may also expand some 2R ops to 3R.
* Rm, Rn => Rm, Rp, Rn

The J_OP_4R prefix will be applicable for operations where Rn is both a source and destination, using Rp to replace the input coming from Rn.

The J_OP_IS prefix will convert normal 3R ops into 3RI.
* It will also convert 2R ops to 2RI (Imm17s, Rn).
* Behavior will typically be the same as the basic instruction.


Example for 2R Imm17s Synthesis (J_OP_IS):
* UUUU-0jjjjj-jjjjjj-1000-WWWWWW-R1-1110 (FF)
* XXXX-ZZZZZZ-Skiiii-ZZZZ-nnnnnn-QY-YYPw (F0)

Witth the Immediate syhesis still following the pattern:
* S-kjjj-jjjj-jjjj-iiii




==== Jumbo-FF with F0 Load/Store (Optional, LDOP) ====

If used with a register-indexed Load/Store:
* UUUU-0iiiii-iiiiii-0000-ssssss-01-1110 (FF)
* XXXX-oooooo-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F0)
** {LDOP_U}.x (Rm, Ro, Disp11u), Rn
* UUUU-1ssiii-iiiiii-0000-ssssss-01-1110 (FF)
* XXXX-oooooo-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F0)
** {LDOP_U}.x (Rm, Ro*Sc, Disp9u), Rn

Where:
* Load will encode a LoadOp instruction on a given type.
** The destination is unchanged, value is loaded into Rn.
* Store will encode an OpStore instruction on a given type.
** The result of the operation will be written to memory.

U Field:
* 0: MOV (Normal Load or Store)
* 1: XCHG (Swap values in Memory and Register)
* 2: ADD    (Mem+Reg)
* 3: SUB_MR (Mem-Reg)
* 4: SUB_RM (Reg-Mem)
* 5: AND    (Mem&Reg)
* 6: OR     (Mem|Reg)
* 7: XOR    (Mem^Reg)
* 8..F: Repeat 0..7, but perform a Volatile operation.

The Rs field will be reserved for now.
* May modify the behavior of OpStore if not ZZR:
** Source value is pulled from Rs rather than Rn.
** Rn is given the prior memory contents.
* This may be used in a way similar to RV AMO ops.


The Volatile operations will be similar to the normal memory accesses, except that the accessed memory may be automatically evicted from the caches to keep memory accesses synchronous between processor cores.

It may be assumed that volatile accesses are synchronous between cores whereas non-volatile accesses may be asynchronous with cores potentially containing memory contents that are out of sync with those in the other cores.


General process for a Volatile access (Example):
* Prior line is flushed regardless of whether it hits or not.
** If dirty (such as from a prior non-volatile access), it is written back.
* Line is fetched from external memory;
* The operation is performed;
* If the operation made the line dirty, it is written back;
* The line's contents are invalidated.

For a Volatile Write, the L1 cache may signal to the outer cache that this line is being written to. In this case, the outer cache may miss on volatile bus requests to this line until the preceding volatile request has completed as a way to enforce memory ordering.

Though, the exact mechanism will be implementation dependent apart from the assumption that volatile accesses will enfornce ordering whereas normal access need not.


==== Jumbo, F1/F2 Block ====

F1/F2
* iiii-iiiiii-iiiiii-jjjj-jjjjXX-X1-1010 (FE)
* XSkk-kkkkkk-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F1/F2)

For the F1/F2 block, a 33-bit immediate will be formed:
* S-jjjj-jjjj-iiii-iiii-iiii-iiii-kkkk-kkkk


For F1 Load/Store and the Jumbo-FE prefix, the MSB of the instruction word will indicate displacement scale:
* 0: Scaled Displacement
* 1: Unscaled Displacement
** This will be N/A for Byte operations.
* For LEA.x (Possible):
** 0: LEA.x
** 1: LEAT.x


For some F2 Block Instructions:

Two Jumbo FE prefixes will give a 64-bit immediate:
* uuuu-uuuuuu-uuuuuu-vvvv-vvvvaa-a1-1010 (FE)
* iiii-iiiiii-iiiiii-jjjj-jjjjbb-b1-1010 (FE)
* cckk-kkkkkk-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F1/F2)

Will give:
* ccaa-abbb-vvvv-vvvv-uuuu-uuuu-uuuu-uuuu (High)
* jjjj-jjjj-iiii-iiii-iiii-iiii-kkkk-kkkk (Low)


If a Jumbo-FF prefix is used with the F1 and F2 blocks, the immediate will be expanded to 18-bits sign-extended.

Generic:
* UUUU-HTTTii-iiiiii-0000-WWWWWW-R1-1110 (FF)
* Sjkk-kkkkkk-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F1/F2)

H=0:
* UUUU-0TTTii-iiiiii-0000-WWWWWW-R1-1110 (FF)
* Sjkk-kkkkkk-mmmmmm-ZZZZ-nnnnnn-QY-YYPw (F1/F2)

For the F1/F2 block, a 18-bit immediate will be formed:
* S-j-iiii-iiii-kkkk-kkkk


H=1:
* UUUU-1bbbbb-bbbbbb-0000-WWWWWW-R1-1110 (FF)
* iiii-iiiiii-mmmmmm-ZZZZ-Sjkkkk-QY-YYPw (F1/F2)

Where, the second immediate (Imm17s) will take the form:
* S-jbbb-bbbb-bbbb-kkkk


The H flag will be special with Store and Branch instructions:
* H=0: Displacement is Extended.
* H=1 (J_OP_4R): Original 10-bit displacement remains in effect.
** Instead, the Rn field is replaced with a second Immediate field (Imm17s).
** This combines with Rn in a similar way to Rm or Ro.
** This will store an immediate to memory, or compare against an immediate.


With the R/U/V/T/W bits as additional opcode or context dependent fields.


==== Jumbo, F8 Block ====

F8
* iiii-iiiiii-iiiiii-XXXX-XXXXXX-X1-1110 (FF)
* kkkk-kkkkkk-kkkkkk-ZZZZ-nnnnnn-SY-YYPw (F8)

For the F8 Block:
* S-iiii-iiii-iiii-iiii-kkkk-kkkk-kkkk-kkkk

Though, for some operations the S bit may instead function as an opcode bit (with a 32-bit immediate).
The additional X bits will be reserved for additional opcode.


F8
* iiii-iiiiii-iiiiii-aaaa-aaaaXX-X1-1010 (FE)
* jjjj-jjjjjj-jjjjjj-bbbb-bbbbXX-X1-1010 (FE)
* kkkk-kkkkkk-kkkkkk-ZZZZ-nnnnnn-SY-YYPw (F8)

Will encode a 64-bit immediate:
* aaaa-aaaa-iiii-iiii-iiii-iiii-bbbb-bbbb
* jjjj-jjjj-jjjj-jjjj-kkkk-kkkk-kkkk-kkkk

Note that the decoding of the register field for the 96-bit F8 Block in XG3 will differ from that in XG2, if interpreted in terms of the repacking rules.

The additional X bits will be reserved for additional opcode.


=== F3 Block (User/Dynamic) ===

The F3 Block is intended primarily for User Extensions.

These may include instructions which only exist for a particular implementation and are not assumed to be portable between implementations, as well as potentially dynamically configurable or bank-swapped instruction blocks.

Likely Split:
* XXXX-oooooo-mmmmmm-0ZZZ-nnnnnn-Q0-1110 (F3)
** Implementation Specific Statically-Assigned Instructions
* XXXX-oooooo-mmmmmm-1ZZZ-nnnnnn-Q0-1110 (F3)
** Dynamically Configured Blocks (DCBs).
** May consist of 8 instruction Blocks (ZZZ) that may be selected from.

A pair of architectural registers may exist to hold the state of the DCBs, which in turn hold the mappings for the decoder to map these blocks into its own internal decoder space.

The management of DCBs falls outside the scope of this specification.

Portable and statically compiled code should not use DCBs.


=== Major Extensions ===

List of Major Extensions:
* ALUX: 128-bit ALU Operations
* BCD: Binary Coded Decimal
* BITMOV: Bitfield Move
* BITNN: Bitfield NN
* DMAC: Multiply Accumulate
* FPIMM: FPU Immediates
* GFP: General Floating Point
* GSV: General SIMD Vectors
* IMMB: Secondary Immed for Store and Bcc
* LDOP: Load/Store Operations
* PRED: Predicated Instructions
* TTAG: Type Tag Instructions

Secondary Derived Extensions:
* GFPX: Extended Binary128 operations.
** Depends on ALUX.
* GSVF: Floating Point SIMD Vectors
** Depends on GSV.
* XMOV: Extended 96-bit Addressing
** Depends on ALUX.
* BCE: Bounds Checked Enforcing (Capabilities)
** Depends on XMOV.
* RGB: Instructions for RGB Pixel / Texel Manipulation
** Depends on GSV
* RGBF: Instructions for Floating Point RGB
** Depends on RGB and GSVF
* LDTEX: Load Texture
** Depends on RGBF
* RGBX: Extended RGB
** Depends on RGBF
* UAB: Compressed Audio Block
** Depends of GSVF

=== ADC ===

*         1001-000010-mmmmmm-0001-nnnnnn-00-0010  ADC		Rm, Rn
*         1001-000010-mmmmmm-0001-nnnnnn-10-0010  ADC.L		Rm, Rn
* J_OP_4R 1001-000010-mmmmmm-0001-nnnnnn-00-0010  ADC		Rm, Ro, Rn
* J_OP_4R 1001-000010-mmmmmm-0001-nnnnnn-10-0010  ADC.L		Rm, Ro, Rn


Add with Carry, Rn=Rn+Rm+SR.T, with SR.T being updated to reflect the carry-out bit.



=== ADDx ===

*        0000-oooooo-mmmmmm-0001-nnnnnn-00-0010  ADD		Rm, Ro, Rn
*        1100-oooooo-mmmmmm-0101-nnnnnn-00-0010  ADDS.L		Rm, Ro, Rn
*        1100-oooooo-mmmmmm-0101-nnnnnn-10-0010  ADDU.L		Rm, Ro, Rn

*        iiii-iiiiii-mmmmmm-0000-nnnnnn-00-1010  ADD		Rm, Imm10u, Rn
*        iiii-iiiiii-mmmmmm-0001-nnnnnn-00-1010  ADD		Rm, Imm10n, Rn
*        iiii-iiiiii-mmmmmm-0011-nnnnnn-00-1010  ADDS.L		Rm, Imm10u, Rn
*        iiii-iiiiii-mmmmmm-0100-nnnnnn-00-1010  ADDS.L		Rm, Imm10n, Rn
*        iiii-iiiiii-mmmmmm-0011-nnnnnn-10-1010  ADDU.L		Rm, Imm10u, Rn
*        iiii-iiiiii-mmmmmm-0100-nnnnnn-10-1010  ADDU.L		Rm, Imm10n, Rn
*        iiii-iiiiii-iiiiii-0010-nnnnnn-01-0010  ADD		Imm16s, Rn

* J_IMM  iiii-iiiiii-mmmmmm-0000-nnnnnn-00-1010  ADD		Rm, Imm33s, Rn
* J_IMM  iiii-iiiiii-mmmmmm-0011-nnnnnn-00-1010  ADDS.L		Rm, Imm33s, Rn
* J_IMM  iiii-iiiiii-mmmmmm-0011-nnnnnn-10-1010  ADDU.L		Rm, Imm33s, Rn

* J2_IMM iiii-iiiiii-mmmmmm-0000-nnnnnn-00-1010  ADD		Rm, Imm64, Rn

* J_OP   iiii-iiiiii-iiiiii-0010-nnnnnn-s1-0010 ? ADD		Imm33s, Rn
* J2_IMM iiii-iiiiii-iiiiii-0010-nnnnnn-s1-0010 ? ADD		Imm64, Rn

Optional (ALUX):
*        0000-oooooo-mmmmmm-0001-nnnnnn-10-0010  ADDX		Xm, Xo, Xn
* J_IMM  0000-oooooo-mmmmmm-0001-nnnnnn-10-0010  ADDX		Xm, Imm33s, Xn



Perform an integer ADD operation.
* ADD: Adds two 64-bit integers.
* ADDX: Adds two 128-bit register pairs (ALUX).
* ADDS.L: Add two integers with 32-bit sign extension.
* ADDU.L: Add two integers with 32-bit zero extension.

For the ADDX instruction, the Even register will hold the low-order bits and the Odd register the high order bits. Carry propagation will happen between the low and high bits, causing a 128-bit integer result to be generated. When an immediate form is used, the value will be sign extended across both registers.

* (ALUX) ANDX: 128-bit AND over register pairs.

*        iiii-iiiiii-mmmmmm-1100-nnnnnn-00-0110  BNTST		Rm, Rn, Disp10s
*        iiii-iiiiii-mmmmmm-1100-nnnnnn-10-0110  BTST		Rm, Rn, Disp10s
*        iiii-iiiiii-mmmmmm-1101-nnnnnn-00-0110  BLT		Rm, Rn, Disp10s
*        iiii-iiiiii-mmmmmm-1101-nnnnnn-10-0110  BGE		Rm, Rn, Disp10s
*        iiii-iiiiii-mmmmmm-1110-nnnnnn-00-0110  BLTU		Rm, Rn, Disp10s
*        iiii-iiiiii-mmmmmm-1110-nnnnnn-10-0110  BGEU		Rm, Rn, Disp10s

*        iiii-iiiiii-mmmmmm-1111-nnnnnn-10-0110  BNE		Rm, Rn, Disp10s

* Compare Rm op Rn and Branch based on the result.
* EQ   : (Rm EQ Rn)    (Equal)
* NE   : (Rm NE Rn)    (Not Equal)
* LT   : (Rm LT Rn)    (Less Than, Signed)
* GE   : (Rm GE Rn)    (Greater or Equal, Signed)
* LTU  : (Rm LTU Rn)   (Less Than, Unsigned)
* GEU  : (Rm GEU Rn)   (Greater or Equal, Unsigned)
* BTST : (Rm AND Rn)   (Test)
* BNTST: !(Rm AND Rn)  (Not-Test)

If the condition is true, the branch is taken, otherwise it is not taken.

** Branches will be relative to the base address of the branch instruction.

=== BITMOV ===

* J_IMM   0010-oooooo-mmmmmm-0010-nnnnnn-00-0010  BITMOVS	Rm, Ro, Rn, Imm24
* J_IMM   0010-oooooo-mmmmmm-0010-nnnnnn-10-0010  BITMOVSX	Xm, Ro, Xn, Imm24
* J_IMM   0011-oooooo-mmmmmm-0010-nnnnnn-00-0010  BITMOV	Rm, Ro, Rn, Imm24
* J_IMM   0011-oooooo-mmmmmm-0010-nnnnnn-10-0010  BITMOVX	Xm, Ro, Xn, Imm24

* J_OP_4R 0010-oooooo-mmmmmm-0010-nnnnnn-00-0010  BITMOVS	Rm, Ro, Rp, Rn
* J_OP_4R 0010-oooooo-mmmmmm-0010-nnnnnn-10-0010  BITMOVSX	Rm, Ro, Rp, Rn
* J_OP_4R 0011-oooooo-mmmmmm-0010-nnnnnn-00-0010  BITMOV	Rm, Ro, Rp, Rn
* J_OP_4R 0011-oooooo-mmmmmm-0010-nnnnnn-10-0010  BITMOVX	Rm, Ro, Rp, Rn


Move a value from one bitfield to another (Optional, BITMOV).

The immediate field from the jumbo prefix is decoded as a single immediate.

The BITMOV instruction will move a bitfield while leaving copying all bits outside the masked region uchanged.

The BITMOVS instruction will selectively invert any bits outside the masked region based on the MSB of the masked value.


Imm24:
* ( 7: 0): Shift to Apply
* (15: 8): Mask Low Bit (L)
* (23:16): Mask High Bit (H)

The value in Rm is shifted and then combined with the value in Ro according to the mask encoded in the L and H fields of the immediate.

In the 4R variant, the value in Rm is shifted and combined with the value from Rp with Ro giving the shift and mask in the same format as the Imm24.

For mask generation:
* L: Generates 1s for bits GE L;
* H: Generates 1s for bits LT H;
* H GT L: Combine using AND
* H LE L: Combine using OR 

The shift will be encoded as a signed 8 bit shift from -63 to 63 for BITMOV or -127 to 127 for BITMOVX. Positive values encode left shifts and negative values encode right shifts.



=== BITNN ===

*         0010-oooooo-mmmmmm-0111-nnnnnn-10-0010  BITNN		Rm, Ro, Rn
* J_OP_4R 0010-oooooo-mmmmmm-0111-nnnnnn-10-0010  BITNN		Rm, Ro, Rp, Rn

Optional (BITNN): Bitwise Neural Net.
* This may be used for weighted bitwise classifier tasks.

Rm:
* Will be 16x 1-bit inputs in 1-bit mode,
* or 16x 2b in 2-bit mode.

Ro:
* (47: 0), Weights, 3 bits each
* (55:48), Bias, 8 bit sign-extended.
* (57:56), Mode
* (   58), Range Flag
* (63:59), Reserved (MBZ)

Rp (4R) / Rn (3R):
* Will be the, 3rd Source, holding output bits.

Rn:
* Will hold the output bits / destination.

Mode:
* 00: 1-bit in, 1-bit out
* 01: 2-bit in, 1 bit out
* 10: 1-bit in, 2 bit out
* 11: 2-bit in, 2-bit out

In 1-bit input mode, each input will be 1 bit:
* 0 => -1, 1 => 1

In 2 bit input mode, each input will be 2 bits:
* 00 => 0
* 01 => 1
* 10 => 0
* 11 => -1

Weights will be 3 bits and defined as:
* 000 => 0
* 001 => 1
* 010 => 3
* 011 => 7
* 100 => 0
* 101 => -1
* 110 => -3
* 111 => -7

Each input will be multiplied by each weight, then the bias will be added, and the final result will be compared against 0.

Output will shift the 3rd input to the left and add the new output into the low-order bit(s).

Output in 1 bit mode will be based on whether the result is greater than or equal to zero.
* 0: Result (after adding bias) was negative.
* 1: Result (after adding bias) was positive or zero.

In 2 bit output mode, the output will be +1 or -1.
* 01: Result is positive.
* 11: Result is negative.

The Range Flag will modify the behavior of bias handling:
* 0: The intermediate result and bias will be added with an 8 bit range.
* 1: The intermediate result and bias will be added with a 9 bit range.

The Range Flag will modify the behavior in the operation in the case where adding the bias would lead to an overflow outside of the 8-bit range. If this flag is set, the range will be expanded such that the overflow will not happen. Otherwise, if Clear, the result will wrap on overflow.

Optional (RGBX), Linear interpolation.

Linearly interpolate between two vectors.

=== BLKUAB ===

* 1100-oooooo-mmmmmm-0110-nnnnnn-00-0010  BLKUAB1		Rm, Ro, Rn
* 1100-oooooo-mmmmmm-0110-nnnnnn-10-0010  BLKUAB2		Rm, Ro, Rn

Optional (UAB), Extract a value from a UAB block.

Extract an audio sample from a block.

These blocks will come in two formats, UAB1 and UAB2.
The UAB1 format will store 16 samples in 32 bits.
The UAB2 format will store 16 samples in 64 bits.

The block is given in Rm, with a sample index in Ro.

The result is unpacked to 16-bit linear PCM and stored in Rn.

For UAB1, the output will be encoded as:
* (15: 0): Sample Value
* (31:16): Sample Value (Repeat)

For UAB2, the output will be encoded as:
* (15: 0): Sample Value (Left)
* (31:16): Sample Value (Right)


UAB1:
* ( 5: 0): Line Start (S.E3.F2)
* (11: 6): Line End   (S.E3.F2)
* (15:12): Line Sigma (  E3.F1)
* (31:16): Sample Selector Bits

UAB2:
* ( 7: 0): Line Start (S.E3.F4)
* (15: 8): Line End   (S.E3.F4)
* (23:16): Line Sigma (Z.E3.F4)
* (31:24): Left/Right Bias (Zero for Mono)
* (63:32): Sample Selector Bits

The block will represent a line segment given as a start and end endpoint.
To decode a sample, linearly interpolate between the start and end value via the sample index.

The Sigma will represent a distance above or below this line segment, which will be either added to or subtracted from the interpolated value (based on 
the selector bits) to get the result.

The line sigma will always be positive for a UAB2 block. The use of a negative Sigma will be reserved, and may be used to escape to another block format.

The Left/Right bias will represent a constant bias which will be added to the left channel and subtracted from the right channel.


Selector bits are organized with the first sample in the low order bits, and the last sample in the high bits.

For each selector, 1 Bit:
* 0: V=Pred-Sigma
* 1: V=Pred+Sigma

For each selector, 2 Bits:
* 00: Small Negative (-0.333 * Sigma)
* 01: Large Negative (-1.000 * Sigma)
* 10: Small Positive (+0.333 * Sigma)
* 11: Large Positive (+1.000 * Sigma)

Or, Alternate:
* 00: Small Negative (-0.375 * Sigma)
* 01: Large Negative (-1.000 * Sigma)
* 10: Small Positive (+0.375 * Sigma)
* 11: Large Positive (+1.000 * Sigma)

Endpoints are encoded as a microfloat value which expands into a 12-bit integer format (PCM):
0: 00000000xxxx
1: 00000001xxxx
2: 0000001xxxx0
3: 000001xxxx00
4: 00001xxxx000
5: 0001xxxx0000
6: 001xxxx00000
7: 01xxxx000000

Negative values will invert the value of the decoded sample.

Interpolation and other calculations will be performed in terms of PCM values.
An encoder should ensure that decoded values will not go out of range.




=== BLKUTX1 ===

* 1100-oooooo-mmmmmm-0010-nnnnnn-00-0010  BLKUTX1		Rm, Ro, Rn

Optional (RGB), Extract pixels from blocks encoded in UTX1.

UTX1 will encode 16 pixels in 32-bits (2bpp).

The Rm field will give the pixel block, and Ro will give an index into this block. Only the low order bits of Ro will be used, with any other bits being ignored.

The output given in Rn will be in unpacked RGBA64 form, with the alpha channel set as FFFF.

For UTX1, the colors endpoints will be given as an RGB444 center, and a 4-bit Delta (YMax-YMin), with 16-bits for pixel data: PPPPDRGB.

The RGB components and delta will be expanded to 8 bits by repeating the nybble.

The Delta will be used to calculate the minimum and maximum colors, and will need to be chosen such that the RGB components do not overflow or underflow.

Where:
* Cm=Cc-(D>>1);		//Minimum
* Cn=Cm+D;			//Maximum
* Repeated for all 3 components.

Within the pixel bits, 0 will select the minimum color, and 1 will select the maximum color.


=== BLKUTX2 ===

* 1100-oooooo-mmmmmm-0010-nnnnnn-10-0010  BLKUTX2		Rm, Ro, Rn

Optional (RGB), Extract pixels from blocks encoded in UTX2.

UTX2 will encode 16 pixels in 64-bits (4bpp).

The Rm field will give the pixel block, and Ro will give an index into this block. Only the low order bits of Ro will be used, with any other bits being ignored.

The output given in Rn will be in unpacked RGBA64 form.

For UTX2, a pair of RGB555A endpoints will be given:
* (15: 0): Color A
* (31:16): Color B
* (63:32): Pixel Bits (4x4, 2bpp)

The alpha bits of the first and second endpoint will encode a mode (A,B):
* 00: Opaque Block, Linear Interpolation.
** Both endpoints will be interpreted as being Opaque.
* 01: Bit-Select Alpha
** Both endpoints will be interpreted as having Alpha.
* 10: Reserved
* 11: Translucent Block, Linear Interpolation.
** Both endpoints will be interpreted as having Alpha.

With linear interpolation, for each 2 bit pixel:
* 00: ColorB
* 01: 2/3*ColorB + 1/3*ColorA
* 10: 1/3*ColorB + 2/3*ColorA
* 11: ColorA

Or, Also Valid:
* 00: ColorB
* 01: 5/8*ColorB + 3/8*ColorA
* 10: 3/8*ColorB + 5/8*ColorA
* 11: ColorA

For interpolated translucent blocks, the alpha will be interpolated along with the color.

For this usage, the difference between 1/3 + 2/3 or 3/8 + 5/8 will be considered close enough that either interpretation may be considered as valid.


For bit select alpha, the high bit will be interpreted as selecting the color, with the low bit selecting the alpha. No interpolation will be performed.

With bit-selection, for each 2 bit pixel:
* 00: ColorB, AlphaB
* 01: ColorB, AlphaA
* 10: ColorA, AlphaB
* 11: ColorA, AlphaA

Pixel ordering will be such that 0 refers to the pixel at the low order bit position, and 15 at the high order bit position. These operations will not care about the internal pixel ordering within the block (which may be either raster or morton).


=== BLKUTX3 ===

* 1001-oooooo-mmmmmm-0110-nnnnnn-10-0010  BLKUTX3L		Xm, Ro, Rn

Optional (RGB), Extract a texel from UTX3.

The UTX format will come in two sub-variants:
** LDR, with RGBA32 endpoints.
** HDR, with with FP8U endpoints in place of linear 8-bit values.

Basic Format:
* ( 31: 0): ColorA
* ( 63:32): ColorB
* ( 95:64): Selectors (RGB)
* (127:96): Selectors (Alpha)

Selectors for RGB and Alpha will be 4x4x2:
* 00: ColorB
* 01: 2/3*ColorB + 1/3*ColorA
* 10: 1/3*ColorB + 2/3*ColorA
* 11: ColorA

Or, Also Valid:
* 00: ColorB
* 01: 5/8*ColorB + 3/8*ColorA
* 10: 3/8*ColorB + 5/8*ColorA
* 11: ColorA


=== BRA/BSR ===

* jjjj-oooooo-mmmmmm-1100-nnnnnn-s0-0010  BRA Disp23
* jjjj-oooooo-mmmmmm-1101-nnnnnn-s0-0010  BSR Disp23
** Disp = { s, nnnnnn, jjjj, oooooo, mmmmmm }
** Multiple of 4 bytes, relative to BasePC.

BRA/BSR encoding:
* In RISC-V terms, BRA will be a "JAL X0, Disp" and BSR "JAL X1, Disp".
** RV JAL may still be used in XG3RV Mode.
** However, branching to a non 32-bit aligned address is undefined.
** Unlike RV JAL, these may not encode additional link registers.
* Branch reach will be +/- 16MB
** This will only be able to branch to a 32-bit aligned address.

Note that BRA and BSR may be considered as subtypes of RV's JAL, trading the flexible link register for more range.

Note that BT and BF may exist as pseudo-instructions represented predicated forms of BRA:
* jjjj-oooooo-mmmmmm-1100-nnnnnn-s0-0000  BT Disp23
* jjjj-oooooo-mmmmmm-1100-nnnnnn-s0-0001  BF Disp23


=== BCDADC ===

*         1000-001000-mmmmmm-0001-nnnnnn-00-0010  BCDADC	Rm, Rn
*         1000-001000-mmmmmm-0001-nnnnnn-10-0010  BCDADCX	Xm, Xn
*         1000-001001-mmmmmm-0001-nnnnnn-00-0010  BCDSBB	Rm, Rn
*         1000-001001-mmmmmm-0001-nnnnnn-10-0010  BCDSBBX	Xm, Xn

* J_OP_4R 1000-001000-mmmmmm-0001-nnnnnn-00-0010  BCDADC	Rm, Rp, Rn
* J_OP_4R 1000-001000-mmmmmm-0001-nnnnnn-10-0010  BCDADCX	Xm, Xp, Xn
* J_OP_4R 1000-001001-mmmmmm-0001-nnnnnn-00-0010  BCDSBB	Rm, Rp, Rn
* J_OP_4R 1000-001001-mmmmmm-0001-nnnnnn-10-0010  BCDSBBX	Xm, Xp, Xn

Optional (BCD): Perform addition or subtraction of Packed BCD numbers.

The SR.T flag is used as a Carry or Borrow flag, and is updated based the result of the leftmost digit.

For this instruction, the 64-bit register is interpreted as a 16-digit BCD number, with each 4 bits corresponding to a decimal digit (0..9). When performing an addition, if the result of adding a pair of digits (plus the carry) is larger than 9, then 6 will be added in this location (so A maps to 0, B to 1, ...), and a carry will be propagated to the next digit.

The SBB case will be similar to ADC, but will perform a 9s complement remapping (0123456789 -> 9876543210) on Rm and will invert the value of SR.T.


These instructions may be daisy-chained to support larger numbers.


=== BNDCHK ===

Optional (TTAG):
* 1000-000000-mmmmmm-0001-nnnnnn-00-0010  BNDCHK.B		Rm, Rn
* 1000-000001-mmmmmm-0001-nnnnnn-00-0010  BNDCHK.W		Rm, Rn
* 1000-000010-mmmmmm-0001-nnnnnn-00-0010  BNDCHK.L		Rm, Rn
* 1000-000011-mmmmmm-0001-nnnnnn-00-0010  BNDCHK.Q		Rm, Rn
* 1000-000100-mmmmmm-0001-nnnnnn-00-0010  BNDCHK		RmImm6u, Rn

Optional (TTAG+PRED):
* 1000-000101-mmmmmm-0001-nnnnnn-00-0010  BNDCMP		RmImm6u, Rn


Optional (TTAG): Check Array Bounds.

The Bounds Check operation will verify whether an index (given) in Rm, falls within the bounds of the array pointer given in Rn.

The immediate forms will give the index to check as a byte offset.


The pointer is assumed to follow the tagged pointer scheme defined in the ABI spec. But, as relevant for these instructions, if the high 4 bits are 0, it is assumed to be a pointer (with the remaining 12 bits for the object type).

Here, a tag with the high 4 bits will be used for a 0011 for a bounded pointer, with other patterns assumed to be non-pointer. The pointer in this case will not encode the type or other properties of the pointer.


The Check operation will determine bounds and, if the bounds check fails, will raise a bounds check exception. It will also raise an exception if given a tagged pointer that does not represent either a raw pointer or a bounded array.


The Compare operation is similar, but will update SR.T based on the result.
SR.T will be set if the access is in-bounds, or cleared if the access is out-of-bounds or is an unhandled type.

Check and Compare will differ regarding the handling of Tagged Object:
* BNDCHK.x will allow access to all Tagged Object types.
* BNDCMP.x will only allow Tagged Object if the tag is 0x0000.


The Bounded Pointer format is:
* Bits (63:60)=Tag (0011)
* Bits (59:56)=Array Bias
* Bits (55:48)=Array Bound
* Bits (47: 0)=Address

The bound is given in bits (55:48) as an E5.F3 microfloat.
* 00:  0  01:  1  02:  2  03:  3  04:  4  05:  5  06:  6  07:  7
* 08:  8  09:  9  0A: 10  0B: 11  0C: 12  0D: 13  0E: 14  0F: 15
* 10: 16  11: 18  12: 20  13: 22  14: 24  15: 26  16: 28  17: 30
* 18: 32  19: 36  1A: 40  1B: 44  1C: 48  1D: 52  1E: 56  1F: 60
* 20: 64  21: 72  22: 80  23: 88  24: 96  25:104  26:112  27:120
* 28:128  29:144  2A:160  2B:176  2C:192  2D:208  2E:224  2F:240
* 30:256  31:288  32:320  33:352  34:384  35:416  36:448  37:480
* 38:512  39:576  3A:640  3B:704  3C:768  3D:832  3E:896  3F:960
* 40:1024 ...
* ...

The Bias is a value with the same exponent as the Bound, but represents a non-normalized value. It is added to the index prior to checking against the bound.


Note that these patterns will not usually be updated directly during pointer arithmetic, but rather adjusted via a LEAT instruction. Note that this adjustment will use the carry/borrow from the low order bits to adjust the Bias.

Also because the bounds are inexact, the bounds for the region will generally be padded out. They are by definition unable to perform precise bounds checks on arrays.


These instructions will be part of the TTAG extension.

* J_IMM  0010-oooooo-mmmmmm-1001-nnnnnn-10-0010  CMPNE.Q	Rm, Imm33s, Rn
* J_IMM  0011-oooooo-mmmmmm-1001-nnnnnn-10-0010  CMPLT.Q	Rm, Imm33s, Rn
* J_IMM  1010-oooooo-mmmmmm-1001-nnnnnn-10-0010  CMPHI.Q	Rm, Imm33s, Rn
* J_IMM  1011-oooooo-mmmmmm-1001-nnnnnn-10-0010  CMPHS.Q	Rm, Imm33s, Rn

Compare two values (Rm and Ro), or compare a value and an immediate, and store the result in Rn.

Comparisions:
* EQ: Equal
* NE: Not Equal
* GT: Greater Than
* LT: Less Than
* GE: Greater Than or Equal
* HI: Unsigned Greater Than
* HS: Unsigned Greater Than or Equal

Notes, the difference between GT/LT and GE/LE is mostly one of switching the operand order.

The Reason for using GT/LT for the Immed cases was because GE/LE can be faked by adding a bias, whereas switching the operand order needs to be handled by the decoder. This inconsistency is a design annoyance, but it is more aesthetic than practical.


Note (Optional, PRED):
* If Rn is R0 / ZERO, these will update SR.T based on the result.
** CMPEQ.Q R0, R0, R0	//Set SR.T (SETT Psuedo-Instruction)
** CMPNE.Q R0, R0, R0	//Clear SR.T (CLRT Psuedo-Instruction)

=== FADD ===

*        1000-oooooo-mmmmmm-0101-nnnnnn-00-0010  FADD		Rm, Ro, Rn
*        1101-oooooo-mmmmmm-0110-nnnnnn-00-0010  FADDG		Rm, Ro, Rn
*        0000-oooooo-mmmmmm-0111-nnnnnn-00-0010  FADDA		Rm, Ro, Rn
* J_OP   1000-oooooo-mmmmmm-0101-nnnnnn-00-0010  FADD		Rm, Ro, Rn, I8rm

Optional (FPUX):
*        1000-oooooo-mmmmmm-0101-nnnnnn-10-0010  FADDX		Xm, Xo, Xn

Optional (FPIMM):
*        1101-oooooo-mmmmmm-0110-nnnnnn-10-0010  FADDG		Rm, Imm6f, Rn
* J_OP_I 1000-oooooo-mmmmmm-0101-nnnnnn-00-0010  FADD		Rm, Imm16f, Rn
* J_IMM  1000-oooooo-mmmmmm-0101-nnnnnn-00-0010  FADD		Rm, Imm32f, Rn
* J_OP   1101-oooooo-mmmmmm-0110-nnnnnn-10-0010  FADDG		Rm, Imm16f, Rn
* J_IMM  1101-oooooo-mmmmmm-0110-nnnnnn-10-0010  FADDG		Rm, Imm32f, Rn


Do a floating point ADD in Binary64 format.
* FADD: Basic form, will be hard-wired for RNE and DAZ/FTZ semantics.
* FADDG: Will use dynamic rounding and traps to emulate IEEE semantics.
* FADDA: Reduced Precision, may do a faster/cheaper version.
** FADDA is to deliver precision at least comparable to Binary32.
** It may still require RNE-like rounding, etc, within this precision.



In the J_OP cases, the I8rm may encode a rounding mode:
* 00: RNE, Round to Nearest
* 01: RTZ, Truncate / Round towards zero.
* 02: RUP, Round toward +Inf
* 03: RDN, Round toward -Inf
* 04: Round, ULP+2, Inexact Status

* 08: Round to Nearest (Single)
* 09: Truncate Towards Zero (Single).
* 0A: Round toward +Inf (Single)
* 0B: Round toward -Inf (Single)

In rounding mode 4, the ULP will be moved upwards by 2 bits.
The low 2 bits of inputs and results will be treated as an inexact status.
If these bits hold 0, the result is exact, otherwise the result is inexact. This state will be sticky.


In the Imm6f case, the value will represent an S.E3.F2 microfloat.
These will be decoded as-if they were first expanded to Binary16 as:
* 000xx: 3000, 3100, 3200, 3300 ( 0.125,  0.156,  0.188,  0.219)
* 001xx: 3400, 3500, 3600, 3700 ( 0.250,  0.313,  0.375,  0.438)
* 010xx: 3800, 3900, 3A00, 3B00 ( 0.500,  0.625,  0.750,  0.875)
* 011xx: 3C00, 3D00, 3E00, 3F00 ( 1.000,  1.250,  1.500,  1.750)
* 100xx: 4000, 4100, 4200, 4300 ( 2.000,  2.500,  3.000,  3.500)
* 101xx: 4400, 4500, 4600, 4700 ( 4.000,  5.000,  6.000,  7.000)
* 110xx: 4800, 4900, 4A00, 4B00 ( 8.000, 10.000, 12.000, 14.000)
* 111xx: 4C00, 4D00, 4E00, 4F00 (16.000, 20.000, 24.000, 28.000)

In the Imm6f cases, the value will be decoded as-if it were first converted to Binary16 and then converted to Binary64 using similar behavior as the FLDCH instruction.

Meanwhile, the Imm16f cases will encode the value as Binary16 directly.
* The extra sign bit in the Imm17s case may specify an fraction packing scheme

Note, however, that the Imm16f case will forbid some value ranges:
* Inf, NaN, Subnormals, also 0.0
* This will be to allow the cheapest possible converter:
** Simply moving the bits around and similar.

May define the Binary16 to Binary64 conversion along the immediate path as:
* { (15:14), (14) ? 6'b000000 : 6'b111111, (13:0), 42'b0 }


If the fraction packing is used for Binary16, the basic format stays the same for the mantissa, but the low 2 bits are understood differently:
* 00: Fill mantissa with a repeating 2 bit pattern.
* 01: Fill mantissa with a repeating 3 bit pattern.
* 10: Fill mantissa with a repeating 4 bit pattern.
* 11: Fill mantissa with a repeating 6 bit pattern.

These bits are taken from the remaining 8 bits of the mantissa, and then organized into a 12-bit chunk which is repeated multiple times, with the lowest chunk being rounded to nearest.


Some encodings may encode an Imm32f. This encoding will basically represent the value as a Binary64 with the low 32 bits cut off (S.E11.F20).
* Bit 32 is 0: Low order bits are filled with 0.
* Bit 32 is 1: Low order bits are filled using a fraction packing scheme.

The 32-bit packing scheme differs:
* (31:4) copied to (63:36)
* (11:4) copied to (35:28)
* (11:4) copied to (27:20)
* (11:4) copied to (19:12)
* (11:4) copied to (11: 4)
* ( 3:0) copied to ( 3: 0)


Support for the bit-packing schemes may be considered as Optional.


Note that if FADD is used with an immediate and with Rm==R0 will behave as a plain constant load.

*        iiii-iiiiii-mmmmmm-0111-000000-10-1010  JALRX	(Xm, Disp10s), ZR
*        iiii-iiiiii-mmmmmm-0111-000001-10-1010  JALRX	(Xm, Disp10s), LR
* J_IMM  iiii-iiiiii-mmmmmm-0111-000000-10-1010  JALRX	(Xm, Disp33s), ZR
* J_IMM  iiii-iiiiii-mmmmmm-0111-000001-10-1010  JALRX	(Xm, Disp33s), LR
* J2_IMM iiii-iiiiii-mmmmmm-0111-000000-10-1010  JALRX	(Xm, Abs64), ZR
* J2_IMM iiii-iiiiii-mmmmmm-0111-000001-10-1010  JALRX	(Xm, Abs64), LR

Perform a computed branch to Rm+Disp10s*4.
* JALR with ZR is a plain Branch.
* JALR with LR sets LR to the address of the following instruction.

Several pseudo instructions will be defined in terms of JALR:
* JMP Rm	=> JALR		(Rm, 0), ZR
* JSR Rm	=> JALR		(Rm, 0), LR
* RTS		=> JALR		(LR, 0), ZR

Note that no other destinations are defined and JALR is aliased with the SHAD.X encoding.

JALRX may exist as part of Addr96/BCE, which will be analogous to JALR except that it may perform a branch through a capability. This instruction may trap if the source and desination are not within the same quadrant.


The Abs64 case will behave similarly to the case for memory ops.
* The Abs64 address will have the same general encoding rules as the Link Register.
** Typically Rm will be R1 in this case (understood as ZR).
* The exact behavior of JALRX Abs64 will be Impementation Defined.
** Most likely it will always trap and the OS can decide what to do with it.


=== LDTEX ===

*       1011-oooooo-mmmmmm-0000-nnnnnn-10-0010  LDTEX	(Rm, Ro), Rn
*       1111-oooooo-mmmmmm-0000-nnnnnn-10-0010  LDTEX2	(Rm, Ro), Rn
* J_OP  1011-oooooo-mmmmmm-0000-nnnnnn-10-0010  LDTEX	(Rm, Ro, Imm11), Rn

optional (LDTEX): Load from Texture.

The Rm gives the base address and format of the texture.

Unlike normal Load/Store instructions, Rm will require a 64-bit alignment.
* Note that alignment will still be required even with 16 or 32-bit texels.

The Ro field gives the ST coords, as a pair of 16.16 fixed-point values.
The ST coordinate values will be interpreted as a texel coordinate.

The result texel will be unpacked into RGBA64 form (4x Int16).
* For an LDR texture format, values are linear 0000..FFFF.
* For an HDR format, values are Binary16.


The Rm field will be interpreted as:
* (47: 0): Texture Base Address
* (51:48): Texture X Size
* (56:52): Texture Combined Size (X+Y)
* (59:57): Block Mode
* (63:60): Pointer Tag

Texture X Size:
* 0: Interpret texture as Morton Order
** This will interleave the S and T bits, using this as the index.
* 1..12: X Stride of 2 .. 4096 pixels.
** Y will be shifted left by this many bits, with X in the low bits.
** Note that the low 2-bits of S and T will be cut off for block textures.

Texture Combined Size:
* Interpreted as the log2 of a bit-mask for the texture index (in texels).
* Note that the mask effectively discards the low 4 bits for block coordinates.
** The complete mask would be applicable for RGBA textures.
** However, LDTEX is not required to mask the low 4 bits of the coordinate.

Block Mode:
* 000: UTX2 (LDR)
* 001: RGBA64 / RGBA64F (raw bits)
* 010: UTX3 (LDR)
* 011: UTX3 (HDR)
* 100: RGB15F / RGB4_E4 (? HDR)
* 101: RGB555A (LDR)
* 110: RGBA8 / RGBA32 (LDR)
* 111: RGBA32_FP8U (HDR)

If a displacement is used, it will be interpreted as additional parameters:
* Disp(2:0): Pixel Selector
** 000: Truncate (Default)
** 001: Round Nearest
** 010: Reserved
** 011: Reserved
** 100: Round S and T downward (Truncate)
** 101: Round S upward, T downward
** 110: Round S downward, T upward
** 111: Round S and T upward


For UTX2, See [BLKUTX2]

For RGB555A, See [RGB Pack/Unpack]

RGB555A:
* 0rrrrrgggggbbbbb, Opaque
* 1rrrraggggabbbba, Translucent (3 bit Alpha)


RGB4_E4:
* (15:12): Exponent
* (11: 8): R
* ( 7: 4): G
* ( 3: 0): B
* This is a joint exponent format with denormal components.


=== LDI / Constant Load ===

*         iiii-iiiiii-iiiiii-0000-nnnnnn-01-0010  LDI		Imm16u, Rn
*         iiii-iiiiii-iiiiii-0001-nnnnnn-01-0010  LDI		Imm16n, Rn
* J_OP    iiii-iiiiii-iiiiii-0000-nnnnnn-s1-0010  LDI		Imm33s, Rn
* J2_IMM  iiii-iiiiii-iiiiii-0000-nnnnnn-s1-0010  LDI		Imm64, Rn

*         iiii-iiiiii-iiiiii-0011-nnnnnn-01-0010  SHORI		Imm16u, Rn
* J_OP    iiii-iiiiii-iiiiii-0011-nnnnnn-01-0010  SHORI32	Imm32u, Rn
* J_OP    iiii-iiiiii-iiiiii-0011-nnnnnn-11-0010  LDIHI32	Imm32u, Rn

*         iiii-iiiiii-iiiiii-0100-nnnnnn-01-0010  FLDCH		Imm16u, Rn
*         iiii-iiiiii-iiiiii-0100-nnnnnn-11-0010  PLDCSW	Imm16u, Rn
* J_OP    iiii-iiiiii-iiiiii-0100-nnnnnn-01-0010  FLDCF		Imm32u, Rn
* J_OP    iiii-iiiiii-iiiiii-0100-nnnnnn-11-0010  PLDCH		Imm32u, Rn

* J_OP    iiii-iiiiii-iiiiii-1100-nnnnnn-01-0010  PLDCM8SH	Imm32u, Rn
* J_OP    iiii-iiiiii-iiiiii-1100-nnnnnn-11-0010  PLDCM8UH	Imm32u, Rn


Load a constant value into a register.
* LDI: Load a simple Integer Constant.
* SHORI: Rn = (Rn << 16) | Imm16u
* SHORI32: Rn = (Rn << 32) | Imm32u
* LDIHI32: Load into high 32 bits.
* FLDCH: Load Binary16 to Binary64
* FLDCF: Load Binary32 to Binary64
* PLDCH: Load 2xBinary16 to 2xBinary32
* PLDCSW: Load Imm16u Repeated 4 times.
* PLDCM8SH (RGBF): 4x E4.M3.S to 4x Binary16
* PLDCM8UH (RGBF): 4x E4.M4 to 4x Binary16

FLDCH:
* Nominally loads a Binary16 value which is converted to Binary64.
* Note: The Imm16u form will forbid Subnormal Numbers and Inf/NaN.
** This will allow for a simple bit-repacking implementation.
** But, forbiding these also allows for using an IEEE aware converter.
** Only values allowed are those where an exact conversion is known.
** Note that the FLDCH Imm16 case will also forbid 0.
*** Loading 0 can be done in other ways.
* If Rn==R2, it encodes a BREAK with an Index.


=== LEA ===

*        0100-oooooo-mmmmmm-0000-nnnnnn-10-0010  LEA.B	(Rm, Ro), Rn
*        0101-oooooo-mmmmmm-0000-nnnnnn-10-0010  LEA.W	(Rm, Ro), Rn
*        0110-oooooo-mmmmmm-0000-nnnnnn-10-0010  LEA.L	(Rm, Ro), Rn
*        0111-oooooo-mmmmmm-0000-nnnnnn-10-0010  LEA.Q	(Rm, Ro), Rn
*        iiii-iiiiii-mmmmmm-0000-nnnnnn-10-0110  LEA.B	(Rm, Disp10s), Rn
*        iiii-iiiiii-mmmmmm-0001-nnnnnn-10-0110  LEA.W	(Rm, Disp10s), Rn
*        iiii-iiiiii-mmmmmm-0010-nnnnnn-10-0110  LEA.L	(Rm, Disp10s), Rn
*        iiii-iiiiii-mmmmmm-0011-nnnnnn-10-0110  LEA.Q	(Rm, Disp10s), Rn
*        iiii-iiiiii-iiiiii-0101-nnnnnn-01-0010  LEA.Q	(GP, Disp16u), Rn
* J_IMM  iiii-iiiiii-mmmmmm-0000-nnnnnn-10-0110  LEA.B	(Rm, Disp33s), Rn

Load Effective Address.
* Type gives the Scale to apply to the Ro or Disp field.
* B/W/L/Q gives a scale of 1/2/4/8.

In the case of LEA, the calculation will only cover the effective address space (nominally 48 bits), with the high order bits of the result being set to 0.


=== LEAT ===

*        0000-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.B		(Rm, Disp6s), Rn
*        0001-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.W		(Rm, Disp6s), Rn
*        0010-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.L		(Rm, Disp6s), Rn
*        0011-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.Q		(Rm, Disp6s), Rn
*        0100-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.B		(Rm, Ro), Rn
*        0101-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.W		(Rm, Ro), Rn
*        0110-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.L		(Rm, Ro), Rn
*        0111-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.Q		(Rm, Ro), Rn
* J1_IMM 0000-oooooo-mmmmmm-0100-nnnnnn-10-0010  LEAT.B		(Rm, Disp33s), Rn

The LEAT instruction will be similar to LEA, except that it will nominally preserve the Type Tag (High Order Bits).

If used with PC as a base register, LEAT will tag the result in the same way as the Link Register, albeit with the U bits and S/T set to Zero.


=== MAC ===

*       0000-oooooo-mmmmmm-0110-nnnnnn-00-0010  MACS.L		Rm, Ro, Rn
*       0001-oooooo-mmmmmm-0110-nnnnnn-00-0010  MACU.L		Rm, Ro, Rn
*       0000-oooooo-mmmmmm-0110-nnnnnn-10-0010  MACS.L		Rm, Imm6u, Rn
*       0001-oooooo-mmmmmm-0110-nnnnnn-10-0010  MACU.L		Rm, Imm6u, Rn
* J_IMM 0000-oooooo-mmmmmm-0110-nnnnnn-10-0010  MACS.L		Rm, Imm33s, Rn
* J_IMM 0001-oooooo-mmmmmm-0110-nnnnnn-10-0010  MACU.L		Rm, Imm33s, Rn

Optional (DMAC): Integer Multiply Accumulate.
* Rn = Rn + Rm * Ro
* Rn = Rn + Rm * Imm

These perform a non-widening multiply accumulate.
* The 32 bit signed or unsigned values are multiplied.
* The result is added to a 64-bit accumultator.

Note that if the multiply overflows, the result will be truncated and extended in the same way as if a 32-bit multiply had overflowed, with the sign or zero extended result then being added to the accumulator.


=== MOD ===

*        0101-oooooo-mmmmmm-0110-nnnnnn-00-0010  MODS.Q		Rm, Ro, Rn
*        0101-oooooo-mmmmmm-0110-nnnnnn-10-0010  MODU.Q		Rm, Ro, Rn
*        0101-oooooo-mmmmmm-0111-nnnnnn-00-0010  MODS.L		Rm, Ro, Rn
*        0101-oooooo-mmmmmm-0111-nnnnnn-10-0010  MODU.L		Rm, Ro, Rn
* J_IMM  0101-oooooo-mmmmmm-0111-nnnnnn-00-0010  MODS.L		Rm, Imm33s, Rn
* J_IMM  0101-oooooo-mmmmmm-0111-nnnnnn-10-0010  MODU.L		Rm, Imm33s, Rn
* J_IMM  0101-oooooo-mmmmmm-0110-nnnnnn-00-0010  MODS.Q		Rm, Imm33s, Rn
* J_IMM  0101-oooooo-mmmmmm-0110-nnnnnn-10-0010  MODU.Q		Rm, Imm33s, Rn


Perform a Modulo operation, dividing two integers and keeping the remainder.

Note: This operation may be potentially be implemented via trap-and-emulate and thus very slow. This operation will be considered optional in hardware but should be provided by emulation in system/firmware code.



== MIN/MAX ==

*       0100-oooooo-mmmmmm-0001-nnnnnn-00-0010  MIN			Rm, Ro, Rn
*       0100-oooooo-mmmmmm-0001-nnnnnn-10-0010  MAX			Rm, Ro, Rn
* J_IMM 0100-oooooo-mmmmmm-0001-nnnnnn-00-0010  MIN			Rm, Ro, Rn

Select the minimum or maximum of the integers given in Rm or Ro, storing result in Rn. These operations will treat the values as signed 64-bit integers.

* J2_IMM iiii-iiiiii-mmmmmm-1010-nnnnnn-10-0110  MOVU.L		(Rm, Abs64), Rn
* J2_IMM iiii-iiiiii-mmmmmm-1011-nnnnnn-00-0110  MOV.Q		(Rm, Abs64), Rn
* J2_IMM iiii-iiiiii-mmmmmm-0101-nnnnnn-10-0110  MOV.X		Xn, (Rm, Abs64)
* J2_IMM iiii-iiiiii-mmmmmm-0111-nnnnnn-10-0110  MOV.X		(Rm, Abs64), Xn

Optional (IMMB):
* J_OP_4R iiii-iiiiii-mmmmmm-0000-nnnnnn-00-0110  MOV.B		I17s, (Rm, Disp10s)
* J_OP_4R iiii-iiiiii-mmmmmm-0001-nnnnnn-00-0110  MOV.W		I17s, (Rm, Disp10s)
* J_OP_4R iiii-iiiiii-mmmmmm-0010-nnnnnn-00-0110  MOV.L		I17s, (Rm, Disp10s)
* J_OP_4R iiii-iiiiii-mmmmmm-0011-nnnnnn-00-0110  MOV.Q		I17s, (Rm, Disp10s)


Load/Store displacement will be scaled by the element size:
* B will have a scale of 1.
* W will have a scale of 2.
* L will have a scale of 4.
* Q will have a scale of 8.
* X will also have a scale of 8.

For Loads:
* MOV variants will sign extend the loaded value to 64 bits.
* MOVU variants will zero extend the loaded value to 64 bits.
* MOVN variants will one extend the loaded value to 64 bits.

Load and Store operations will ignore the high 16 bits of the address register, which may be used as tag bits.

Nominally, all load/store operations (excluding X) are to be unaligned safe.
* An implementation should at least provide for reasonably fast unaligned access for any of the smaller types.

The X type will perform a Load/Store Pair.
* It is not required to support unaligned access in hardware.
* So, a misaligned MOV.X may be allowed to fault or otherwise be very slow.
* For Unaligned Safe Load/Store Pair, it is better to use a pair of MOV.Q instructions.

The Abs64 mode will be encoded the same as a Disp64, but will have some rules of its own:
* The Immediate will be unscaled and will canonically Absolute;
** The high 16 bits are MBZ for Load/Store ops in a 48-bit VAS.
* The Rm register will be treated as an unscaled offset added to the address.
** Implicitly, behavior will be undefined if Rm is outside +/- 4GB.
** If R1 is used for Rm, it is understood as Zero Register.
*** This will be a typical use-case for Abs64 addressing.
** The use of R0, R2, and R3 will be reserved.


For the IMMB encodings, a 17-bit sign extended value is stored to memory. Note that for 8 and 16 bit types, the Immediate should not be used to encode values outside the nominal range of the type (out of range values will be reserved).


=== MOVxD ===

* 1000-oooooo-mmmmmm-0010-nnnnnn-00-0010  MOVHD		Rm, Ro, Rn
* 1000-oooooo-mmmmmm-0010-nnnnnn-10-0010  MOVLD		Rm, Ro, Rn
* 1001-oooooo-mmmmmm-0010-nnnnnn-00-0010  MOVHLD	Rm, Ro, Rn
* 1001-oooooo-mmmmmm-0010-nnnnnn-10-0010  MOVLHD	Rm, Ro, Rn

Compose a new 64b value with the low or high 32b halves of each register.
* MOVLD: Select Low halves of each register.
* MOVHD: Select High halves of each register.
* MOVHLD: Select High and Low halves.
* MOVLHD: Select Low and High halves.

Here, Rm supplies the value for the high 32 bits of the destination, whereas Ro supplies the value for the low 32 bits of the destination.


=== MOVxW ===

** 0011-oooooo-mmmmmm-0111-nnnnnn-00-0010  MOVHW		Rm, Ro, Rn
** 0011-oooooo-mmmmmm-0111-nnnnnn-10-0010  MOVLW		Rm, Ro, Rn

Compose a new 32b value with the low or high 16b values of each register.
* MOVLD: Select Low words of each register.
* MOVHD: Select High words of each register.
* MOVHLD: Select High and Low words.
* MOVLHD: Select Low and High words.

These instructions operate solely within the low 32 bits of each register, ignoring the high bits on input, and setting the high 32 bits of the result to zero.


=== MOVST ===

* 1110-001111-mmmmmm-0001-nnnnnn-00-0010 ? MOVST	Rm, Rn
* 1110-001111-mmmmmm-0001-nnnnnn-10-0010 ? XMOVST	Xm, Xn

Optional (TTAG): Move a value while setting tag bits to a sign extension of bit 47.


=== MOVTT ===

*       0001-oooooo-mmmmmm-0011-nnnnnn-00-0010  MOVTT		Rm, Ro, Rn
*       0001-oooooo-mmmmmm-0011-nnnnnn-10-0010  MOVTT		Rm, Imm6u, Rn
* J_OP  0001-oooooo-mmmmmm-0011-nnnnnn-10-0010  MOVTT		Rm, Imm16u, Rn

Optional (TTAG): Move a value into the Type Tag (high 16 bits).

The 3R form will take the low 48 bits from Rm and the high 16 bits from the low 16 bits of Ro.

The Imm6 form will encode a bit-pattern for the value to fill, replacing the top 1-4 bits.

In the 3RI Imm6 form, the tag is twiddled:
* 0zzzz0: Rn(63:60)=Imm(4:1)
* 0zzz01: Rn(63:61)=Imm(4:2)
* 0zz011: Rn(63:62)=Imm(4:3)
* 0z0111: Rn(63   )=Imm(4  )

=== RGB Pack/Unpack ===

*       1110-000010-mmmmmm-0001-nnnnnn-00-0010  RGB5PCK32	Rm, Rn
*       1110-000010-mmmmmm-0001-nnnnnn-10-0010  RGB5PCK64	Rm, Rn

Pack will convert from a wider format to a narrower one.
Unpack will convert from a narrower format to a wider one.

RGB555A:
* 0rrrrrgggggbbbbb, Opaque
* 1rrrraggggabbbba, Translucent (3 bit Alpha)

Unpacking to RGB555A to RGB32 will mirror the high bits of each component into the low bits, so 43210 is expanded to 8 bits as 43210432. Note that the alpha-blended case will unpack RGB the same as in the opaque case, so 4321a becomes 4321a432.

Unpacking RGB555A to RGBA64 will behave as if the value were first unpacked to RGBA32 and then from RGBA32 to RGBA64.

Packing will do the reverse process, keeping only the high order bits and discarding the rest. When packing to RGB555A, if the high 4 bits of Alpha are Set, then the pixel will be treated as opaque (and MSB will be clear), otherwise the MSB will be set and the LSB of each component will contain the one of the high 3 bits of the Alpha channel.

* yyyy00cc
* 0000yyyy (Grayscale, All levels)

Where yyyy gives a luma, and irgb/cc gives chroma.
* irgb: Gives 16 colors, where i gives the saturation.
** 0rgb: High saturation (Hi=Y, Lo=1/4 Y)
** 1rgb: Low saturation (Hi=Y, Lo=5/8 Y)
** 0000: Grayscale
** 0111: Orange	(1/1, 5/8, 3/8)
** 1000: Olive	(5/8, 1/1, 3/8)
** 1111: Azure	(3/8, 5/8, 1/1)
* cc: Gives off-white colors:
** 00: RGBI (yyyy interpreted as a traditional 16-color palette)
** 01: Off-White
*** ( 1.. 5)=Magenta (11..15)
*** ( 6..10)=Yellow (11..15)
*** (11..15)=Cyan (11..15)
** 10: Reserved
** 11: Reserved

The UPCK variant does the reverse, mapping this fixed palette back to an RGB555 value.


=== RGB5SHR1 (RGB) ===

*** 1110-000000-mmmmmm-0001-nnnnnn-00-0010  RGB5SHR1	Rm, Rn

Shift components in a packed RGB555 vector right by 1 bit.
Effectively, this is a combined 1-bit shift right operation and a bit-mask.


=== RGB5MINMAX (RGB?) ===

*** 1110-000000-mmmmmm-0001-nnnnnn-10-0010 ? RGB5MINMAX	Rm, Rn

Selects the Minimum and Maximum values from a vector of RGB555 values based on the approximate luminance of each color.

This will select the minimum and maximum values from within the 4 RGB555 colors given in Rm, storing the result into Rn.

Within the result:
* (63:32): Reserved / Zeros
* (31:16): Minimum
* (15: 0): Maximum

The luminance function will depend on the implementation, but for example may be defined in terms of bit-interleving, say:
* G4, G3, R4, B4, G2, R3, B3, G1

Where, one may choose bit interleaving due to having less latency than converting 

The selected values will be implementation dependent in cases where multiple pixels have the same luminance.



=== RGB5CCENC ===

* 0110-oooooo-mmmmmm-0111-nnnnnn-00-0010 ? RGB5CCENC	Rm, Ro, Rn


Color-Cell Encode Helper (Optional, CCENC).
* Rm will give the pixel values to encode.
* Ro will give the Min and Max color endpoints.
* Rn will give both the source and destination for the selector bits.

This operation will classify the pixel colors relative to the endpoints, generating a selector between 00 (Minimum) and 11 (Maximum).
* Each of the 4 colors will be compared relative to the min and max colors.
* 00=Min, 01=5/8 Min + 3/8 Max, 10=3/8 Min + 5/8 Max, 11=Max

The low order 32 bits of Rn will be shifted left by 8 bits, with the new endpoint selectors being added to bits (7:0).


Along with RGB5MINMAX, this may be used to implement a faster color-cell encoder from RGB555.

=== SBB ===

*          1001-000011-mmmmmm-0001-nnnnnn-00-0010 ? SBB		Rm, Rn

=== SETTRAP ===

* 1001-001010-mmmmmm-0001-nnnnnn-10-0010  SETTRAP		Rn, (Rm)

Optional Debug Feature: Set Tripwire Mode
This modifies the status of the pointed-to cache line, such as setting it to be temporarily write protected.

The Rm pattern will encode the desired mode:
* (3:1): Properties to modify (RXW)
* (  0): Set (Disable Access) or Clear (Enable Access)

Accessing the cache line in a way inconsistent with the current mode may cause a fault.

Setting a tripwire may also optionally trap if an attempt is made to address across the tripwire for sake of a given access type, treating it as a memory boundary.


Note that this feature is optional and intended for debugging, as such:
* It may be treated as NO-OP;
* The effects may be transient and forgotten at any point.
* It may only be used disallow access to a piece of memory.


=== SNIPExx ===

* 1000-001011-mmmmmm-0001-nnnnnn-00-0010  SNIPEDC		Rm, Rn
* 1000-001011-mmmmmm-0001-nnnnnn-10-0010  SNIPEIC		Rm, Rn

Optional (System): Calculate Cache Sniping Address.

For the address given in Rm, calculate a "sniper" address within the ZERO or RTS pages. These instructions assume either physical memory mapping, or that these pages are identity mapped.

Accessing the address returned by these SNIPEDC will evict whatever was in the cache-line pointed to by Rm from the L1 D-Cache.

Calling to the address returned by SNIPEIC will evict whatever was in this location in the L1 I-Cache.

These operations will only work for direct-mapped caches.


The SNIPExx instructions will generate a NULL result if SNIPExx is not valid (such as due to using an associative cache). In this case, an INVDC or INVIC cache-flush process will be required instead.

=== SHAD ===

* 0100-oooooo-mmmmmm-0101-nnnnnn-00-0010  SHAD.L		Rm, Ro, Rn
* 0100-oooooo-mmmmmm-0101-nnnnnn-10-0010  SHAD.Q		Rm, Ro, Rn
* 00ii-iiiiii-mmmmmm-1000-nnnnnn-00-1010  SHAD.L		Rm, Imm8s, Rn
* 00ii-iiiiii-mmmmmm-1000-nnnnnn-10-1010  SHAD.Q		Rm, Imm8s, Rn

Optional (ALUX):
* 0100-oooooo-mmmmmm-0011-nnnnnn-10-0010  SHAD.X		Xm, Ro, Xn
* 00ii-iiiiii-mmmmmm-0110-nnnnnn-10-1010  SHAD.X		Xm, Imm8s, Xn

Optional (SHFUN):
* J_OP_4R  0100-oooooo-mmmmmm-0101-nnnnnn-10-0010  SHAD.Q	Rm, Ro, Rp, Rn
* J_OP_4R  00ii-iiiiii-mmmmmm-1000-nnnnnn-10-1010  SHAD.Q	Rm, Imm8s, Rp, Rn


Shift Left Arithmetic.
* SHAD.L will Sign-Extend the Result.
* Negative Shifts will perform a Right Shift Arithmetic.
* SHAD.X will perform a 128 bit shiift.

** The SHAD.X encoding will not exist for R0..R3

Funnel Shift (Optional):
* Perform a funnel shift with an extra input.
* Rather than shifting in zeros, this will shift in bits from Rp.
** Except in Imm8s case if sign is negative, will still pull bits from Rp.

=== SHLD ===

* 0101-oooooo-mmmmmm-0101-nnnnnn-00-0010  SHLD.L		Rm, Ro, Rn

* 00ii-iiiiii-mmmmmm-1001-nnnnnn-10-1010  SHLD.Q		Rm, Imm8s, Rn

Optional (ALUX):
* 0101-oooooo-mmmmmm-0011-nnnnnn-10-0010  SHLD.X		Xm, Ro, Xn
* 00ii-iiiiii-mmmmmm-0111-nnnnnn-10-1010  SHLD.X		Xm, Imm8s, Xn

* J_OP_4R  0101-oooooo-mmmmmm-0101-nnnnnn-10-0010  SHLD.Q	Rm, Ro, Rp, Rn

=== SHLR ===

* 0011-oooooo-mmmmmm-0010-nnnnnn-00-0010  SHLR.L		Rm, Ro, Rn
* 0011-oooooo-mmmmmm-0010-nnnnnn-10-0010  SHLR.Q		Rm, Ro, Rn

Optional (ALUX):
* 0011-oooooo-mmmmmm-0011-nnnnnn-10-0010  SHLR.X		Xm, Xo, Xn

Optional (SHFUN):
* J_OP_4R  0011-oooooo-mmmmmm-0010-nnnnnn-10-0010  SHLR.Q		Rm, Ro, Rp, Rn

Shift Right Logical.
* Negative Shifts will perform a Left Shift.
* SHLR.L will Zero-Extend the Result.

Funnel Shift (Optional):
* Perform a funnel shift with an extra input.
* Rather than shifting in sign extension, this will shift in bits from Rp.
* Will differ from SHAR.Q in that the shift direction will be reversed.

Note that the direction reversion is to aid in forming generic extended precision left shifts; with SHLD.Q-4R dealing with the high bits and SHLR.Q-4R dealing with the low-order bits. This case will also allow for a right shift when the sign is negative.

Note that (unlike the ALUX cases), the funnel shift operators will be limited to 64 bits. The higher order bits of a multi-precision shift will need to be dealt with via MUXing.


=== TST ===

*        0001-oooooo-mmmmmm-1000-nnnnnn-10-0010  TSTN.Q		Rm, Ro, Rn
*        0010-oooooo-mmmmmm-1000-nnnnnn-10-0010  TST.Q		Rm, Ro, Rn
* J_IMM  0001-oooooo-mmmmmm-1000-nnnnnn-10-0010  TSTN.Q		Rm, Imm33s, Rn
* J_IMM  0010-oooooo-mmmmmm-1000-nnnnnn-10-0010  TST.Q		Rm, Imm33s, Rn

Bit-Test.
Set or Clear Rn based on the result of a Bit-Test operation:
* If Rn is set to R0, this will direct the output to SR.T.
* TSTN: Rn = !(Rm & Ro)
* TST: Rn = (Rm & Ro)


=== VSKG / VSKC ==

* 1000-000110-mmmmmm-0001-nnnnnn-00-0010  VSKG		Rm, Rn
* 1000-000111-mmmmmm-0001-nnnnnn-00-0010  VSKC		Rm, Rn

These instructions will Generate or Check a Canary value.
The Generate operation will transform a reference value into an internal form via combining it with a randomized number. The contents of the generated number will be implementation defined.

The Check operation will verify that the values match, and will raise a Canary Check exception if the values don't match. The reference value supplied to the check will be required to be the same as that supplied to the generate instruction.

The Rm value will supply a reference value (magic number), with Rn holding the Canary value.


=== XBLESS ===

* 1110-001110-mmmmmm-0001-nnnnnn-10-0010  XBLESS		Xm, Xn

Optional (BCE): Bless a Capability.

This instruction has behavior similar to a MOVX instruction, except that while moving the value in the registers, it will also set the tag bits to mark the register contents as a capability.

This is a priveleged instruction within the Bounds Check Enforcing extension.

If Bounds Check Enforce is not set, or the processor is in Supervisor Mode, the BLESS operation will always succeed.

If in User Mode and BCE is enabled, this will fault if Xm is not already blessed as a capability (otherwise, behavior will be equivalent to MOVX).
